STEP 1 - IMPLEMENT:
- Read the plan's Overview and Context sections to understand the work
- Implement ALL items in the current Task section (all [ ] checkboxes under it)

STEP 2 - SELF-REVIEW (before running tests):
For each file you modified, re-read your code and check:
- Side effects: if a function sends email/SMS/webhook, does it also update the tracking timestamp/status field?
- Null safety: can any chained property access crash? Add optional chaining or early returns for nullable fields
- Event wiring: if backend emits a new WS/event, is there a listener on the other side (UI subscription, @OnEvent handler)?
- Race conditions: can two concurrent calls corrupt state? Use atomic updates (updateMany with WHERE conditions) for critical paths
- Timezone: are all date/cron operations in the correct timezone (not UTC by default)?
- Domain filters: does the query filter by status/type to exclude cancelled/expired/irrelevant records?
- Public entry points: if a cross-module import targets a module with `public.ts`, route it through `@/{module}/public`
- Layering: do not introduce `adapter -> orchestration`, `orchestration -> adapter`, or `adapter -> adapter`
- Tx-aware outbox: if a Prisma transaction writes outbox records, use `enqueueInTx()` in the transaction and `emitEnqueued()` after commit
- Docs accuracy: if you added/removed models, controllers, or env vars, run `cd api && npm run lint:docs` and fix any failures

Fix any issues found BEFORE running tests. This prevents bugs from reaching the review cycle.

STEP 3 - VALIDATE:
- Run the test and lint commands specified in the plan (e.g., "cargo test", "go test ./...", etc.)
- If the current task changed `api/src/**`, run `cd api && npm run lint:arch:diff -- --base {{DEFAULT_BRANCH}}`
- Fix any failures, repeat until all validation passes
