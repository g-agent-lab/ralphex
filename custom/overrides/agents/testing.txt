## Test Existence and Coverage

1. Missing tests - new code paths without corresponding tests
2. Untested error paths - error conditions not verified
3. Coverage gaps - functions or branches without test coverage
4. Integration test needs - system boundaries requiring integration tests
5. Validation workflow gaps - backend tasks that change `api/src/**` but do not run the architecture guard

## Test Quality

1. Tests verify behavior, not implementation details
2. Each test is independent, can run in any order
3. Descriptive test names that explain what is being tested
4. Both success and error paths tested
5. Edge cases and boundary conditions covered

## Fake Test Detection

Watch for tests that don't actually verify code:
- Tests that always pass regardless of code changes
- Tests checking hardcoded values instead of actual output
- Tests verifying mock behavior instead of code using the mock
- Ignored errors with _ or empty error checks
- Conditional assertions that always pass
- Commented out failing test cases

## Assertion Quality

Check that assertions actually prove the code works:
- Assertions test the RESULT, not just that the mock was called (toHaveBeenCalledWith is necessary but not sufficient — also verify the return value or side effect)
- Error path tests verify the specific error type/message, not just "it throws"
- Tests for stateful operations verify the FINAL STATE (e.g., check DB record, not just that service was called)
- Array/list assertions check content, not just .length (a test checking `result.length === 3` passes even if the items are wrong)
- Async tests properly await and don't silently swallow rejections

## Weak Assertion Detection (Regression-Blind Tests)

Watch for assertions that pass but don't catch format/structure regressions:
- `toContain(substring)` on structured output - passes even if the output is wrapped in JSON, has extra fields, or changed format. Use `toBe()` or `toEqual()` for exact format verification
- `expect(obj).toMatchObject(partial)` without checking that unexpected fields are absent — use `toEqual()` when full shape matters
- `expect(prompt).toContain('some text')` for prompt format tests — doesn't catch if surrounding format changed (e.g., `"action: completed text"` vs `"action: completed {\"result\":\"text\"}"`)
- `toHaveBeenCalledWith(expect.objectContaining({...}))` that checks some fields but misses critical ones — if a new field was added to the contract, verify it's included in existing assertions
- Negative assertions missing: when asserting what output SHOULD contain, also assert what it should NOT contain (e.g., `expect(output).not.toContain('{"result":')` to prevent JSON wrapping)

## Test Independence

1. No shared mutable state between tests
2. Proper setup and teardown
3. No order dependencies between tests
4. Resources properly cleaned up

## Edge Case Coverage

1. Empty inputs and collections
2. Null/nil values
3. Boundary values (zero, max, min)
4. Concurrent access scenarios
5. Timeout and cancellation handling
